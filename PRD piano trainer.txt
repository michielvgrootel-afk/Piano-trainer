Context
The user wants to build a "Guitar Hero for Piano" — an Electron desktop app that connects to any MIDI keyboard, lets users upload sheet music (including PDFs), and provides a falling-notes tutorial with scoring and progressive difficulty. This is a new project built at piano-trainer/.
Problem: Learning piano from sheet music is hard. There's no easy, plug-and-play tool that combines real MIDI keyboard input with Guitar Hero-style visual guidance and gamification.
Outcome: A desktop Electron app where users connect a MIDI keyboard, upload songs (MIDI, MusicXML, or PDF sheet music), and play along with falling notes, getting scored on accuracy. PDFs are converted to playable note data via OMR (Audiveris).

	PDF/OMR Strategy: Audiveris is a mature open-source OMR engine that converts PDF/image sheet music into MusicXML. We bundle the Audiveris JAR + a portable JRE (e.g., Eclipse Temurin JRE, ~50MB compressed) inside the Electron app so the user never needs to install Java themselves. The main process spawns Audiveris as a child process. Flow: PDF → Audiveris CLI → MusicXML → our MusicXML parser → NoteSequence → falling notes. True plug-and-play — drop a PDF, get playable notes.

Phase 6: Tutorials + Gamification
Goal: Built-in progressive lessons with chord display.
Create:
* public/tutorials/*.json — 15-20 lessons (beginner to expert)
* src/data/tutorial-loader.ts
* src/ui/screens/tutorial-select.ts — difficulty tree with unlock system
* src/ui/components/chord-display.ts + src/utils/chord-detector.ts
* Stars (1-3) based on score. Prerequisites gate harder lessons.
Tutorial Progression:
* Level 1: Middle C, basic notes, first chord
* Level 2: Full octave, both hands, eighth notes
* Level 3: Scales, minor chords, simple melodies
* Level 4: Arpeggios, two-hand independence
* Level 5: Complex rhythms, full songs
Phase 7: PDF/OMR Support (Bundled Audiveris)
Goal: Upload a PDF of sheet music and have it automatically converted to playable notes. Zero setup for the user.
Create:
* resources/jre/ — bundled portable JRE (Eclipse Temurin, ~50MB compressed, platform-specific)
* resources/audiveris/ — Audiveris JAR + dependencies
* src/core/omr-bridge.ts — renderer-side IPC calls to trigger Audiveris
* IPC handler in electron/main.ts — spawns bundled JRE + Audiveris CLI: <bundled-jre>/bin/java -jar <bundled-audiveris>.jar -batch -export -output <tmpDir> <input.pdf>
* Progress UI — show "Converting PDF..." with progress spinner while Audiveris runs (can take 10-30s)
* Error handling — if bundled JRE is corrupted/missing, show re-install prompt
* Update file-upload.ts to accept .pdf, .png, .jpg in addition to .mid, .musicxml
* electron-builder extraResources config to include JRE + Audiveris in the packaged app
Flow: User drops PDF → main process runs bundled Audiveris → produces MusicXML in temp dir → renderer reads MusicXML → existing parser pipeline takes over. No Java installation required.
Phase 8: Polish
* 3-2-1 countdown before songs
* Note hit animations (glow/particles)
* Settings screen (MIDI device, latency calibration)
* electron-builder config for packaging/distribution (with bundled JRE + Audiveris as extraResources)
* Auto-updater setup

Key Technical Notes
1. Synchronization: All modules (Canvas, audio, OSMD cursor, input matcher) read from a single clock in timing.ts. This prevents drift.
2. Hit window: +/- 200ms default, adjustable by difficulty. Notes outside the window = missed.
3. Canvas performance: Only render notes within [currentTime - 1s, currentTime + visibleWindow]. Culling keeps it under ~100 notes/frame.
4. Audio latency: Settings screen includes a calibration step to measure round-trip latency and offset input timestamps.
5. Chord detection: When 3+ simultaneous notes match a known chord pattern, display the chord name (e.g., "Cmaj", "Am7").

Verification Plan
1. Phase 1: npm run dev launches Electron window. Connect MIDI keyboard, verify keys light up on virtual keyboard. Confirm falling notes render smoothly at 60fps.
2. Phase 2: Upload a .mid file, verify audio plays and notes fall in sync. Change speed, verify sync is maintained.
3. Phase 3: Play along with a song, verify hits/misses are detected correctly. Check results screen shows accurate score.
4. Phase 4: Upload .musicxml, verify OSMD renders notation and cursor moves in sync with falling notes.
5. Phase 5: Upload a song, close and reopen the app, verify song persists.
6. Phase 6: Complete a tutorial, verify it shows as completed, and the next lesson unlocks.
7. Phase 7: Upload a PDF of sheet music, verify Audiveris converts it and falling notes appear. Test error case (no Java installed) shows helpful message.
8. Phase 8: Run npm run build to produce distributable. Verify packaged app launches and works standalone.
